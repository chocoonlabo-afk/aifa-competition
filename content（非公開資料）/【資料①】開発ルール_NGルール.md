# 【資料①】AIMC 開発ルール・NGルール（修正版）

---

# 1. 基本原則

- フォルダ構成・ルーティングの揺れを無くす
- constants・i18n・SKU・テーブル名は一元管理（文字列直書き禁止）
- 勝手な仕様変更／キー増殖／UI変更は禁止
- コード提出前に必ず 4大NG 自己チェック

---

# 2. スレ運用

- スレ冒頭に目的／ゴール／禁止事項
- 仕様はスレ上ではなく資料化する
- ダブルチェック後のみ確定仕様として扱う

---

# 3. フォルダ構成・ファイル追加

- 現状のフォルダ構成が唯一の正解
- 新規ファイルは事前承認必須
- 破損や揺れがあれば即ストップ

---

# 4. constantsルール（新追記）

**重要方針：**

> constantsは 1ファイルではなく **1フォルダ一元管理**  
> 詳細運用は **constants運用ルール.md** に従う

### 絶対禁止：

- JSX内に文言直書き
- URL直書き
- Supabaseテーブル名直書き
- SKU直書き
- 年度直書き
- importパス揺れ

### import形式：

```
import { ROUTE_APPLY } from '@/constants/routes'
```

### 追加／修正：

- 必ず資料化 → ダブルチェック → 承認

📎 **詳細運用はこちら：**  
`/content/【資料⑤】constants運用ルール.md`

---

# 5. i18n運用

- 全画面日英必須
- JSXに生文字禁止
- i18nキーは事前洗い出し
- キー定義は **constants/i18n.ts**
- メール文面も i18n対象（本文はテンプレ資料）

---

# 6. Stripe連携

- SKU・商品・Webhookは constants に一元管理
- 決済文言・金額は JSX直書き禁止
- Stripeメールテンプレは資料に一元管理

---

# 7. Supabase連携

- TABLE名・Edge Function名・RPC名は constants 一元管理
- Export/Importは CSV 全上書き（差分禁止）
- 勝手な API 想像禁止

---

# 8. OLDフォルダ管理（追記）

- OLDは自動退避禁止  
- 追加時は資料化＋承認
- 削除時は二重承認
- 復元可能性を常に保持

---

# 9. ChatGPT コーディング 4大NG

1. 構文エラーなし（全文確認）
2. 存在しないパス参照禁止
3. 旧版機能の勝手な削除禁止
4. UI変更禁止（理由と承認必須）

---

# 10. コード提出プロセス

提出前：

- 理解メモ
- import依存関係
- 保存処理の流れ
- パス整合性
- UI変更有無（原則NG）

提出時：

- **ファイル全文**
- **4大NGチェック**
- **constants違反チェック**

---

# 11. 成果物の品質線引き

- スレ会話＝仕様 は禁止
- 「資料 → ダブルチェック → 確定」 のみ有効
- 仕様が曖昧なままコーディング禁止
